The only surprise I had was using the HiveMQ-MQTT-Client Java library. I originally used the latest 1.3.4 version with the MQTT 5 support. It worked very well at home with my simulator and my local JMRI server. Then I deployed it at Randall and two issues appeared:

  • hivemq-mqtt-client 1.3.4 brings in a Kotlin 1.9 runtime dependency that is somehow conflicting when loaded dynamically via Jython in JMRI 5.0 -- it conflicts because my own engine has a Kotlin 1.6 dependency and I guess I can‚Äôt have both runtimes loaded in the JVM at the same time? Just a guess. I do my tests at home against the latest JMRI 5.5, so maybe I need to update my production environment to match. That was solved by checking the release dates for HiveMQ -- I simply went back to hivemq-mqtt-client 1.3.1, which was published before Kotling 1.9, thus I know it can only have a Kotlin 1.6 dependency at most. Eventually I want to solve that and update to Kotlin 1.9.
  • HiveMQ has both MQTT 3 and 5 support as a client. I started with MQTT 5. It worked fine at home with Mosquitto 2.0.18. When I deployed at the museum, the calls were rejected by the local Mosquitto broker as ‚Äúwrong version‚Äù. The museum version is Debian‚Äôs Mosquitto 1.5.7, which turns out to only have up to protocol 3.1.1 support. I simply updated the Conductor 2 usage of HiveMQ to switch to the MQTT 3 protocol.

Overall, I‚Äôm satisfied with the implementation. It‚Äôs simple. Usage on the script side is fairly straightforward and clean.
